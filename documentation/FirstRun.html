First Run Guide

    Just before you exit this guide let me take two minutes and explain it's purpose. For those of you new to Linux or migrating from other Linux distributions it might be worth while poking around to find out what we do differently for Atomic Linux to stand out as a unique distribution. Our latest revision will include many tools that you may already be familiar with however I'm sure that being an Atomican you'll not only want to know the OS inside and out but you'll also feel the urge to tweak it, change it or just blow it all away!

    Migrating from windows

      If you're migrating from the ever present Windows flavour of operating systems do not despair at the lack of GUI with which you are presented for all will soon become clear. Linux has a GUI! Most commonly found is XFree86 and is included in Atomix. Linux does not restrict you to applications that will only run in a GUI. This is an important difference in paradigm to Windows offerings. Essentially though Linux won't leave you stranded for applications and tools if you choose not to use a GUI.

    Fundamentals of the console

      For those of you wondering, the 'ALT' stands for alternate and 'CTRL' also means control and I'm still yet to find an Anykey! Anywho the 'local' service is running 4 terminals on your Linux system right now! “Where are they?” I hear you ask. The answer to that lies in the mystery that is the Linux kernel. Essentially because Linux is a multiuser system, more that one user can be logged in at any one time. This can be done in a variety of ways but the two most common are at the console and over TCP/IP.

      For those of you who are rather attached to a mouse this may be a new experience for you. Anyone who has ever had experience with any of the DOS flavours will relish in this new environment. Getting around in the shell (Bash) is easy. Type ls to get a current listing of the current directory. Typing cd will Change Directory but wait a minute don't go quiting this program just yet for I'm about to demonstrate one of the most powerful features of Linux, The alternate console!

      By using a combination of the 'ALT' key and the function keys 1-4 you can cycle through the first four console logins. Pressing 'ALT' + 'F1' will return you here. On each of these new consoles you will be presented with another login. For now login on an alternate console, then you can return here so that you can continue to read this document and play with your system. Stop sniggering I said system! Read on and learn how to navigate your home directory and create, modify and delete stuff!

      As mentioned ls lists a the current directory and cd will change the directory and these are fundamental commands for navigating the filesystem. Let's now have a quick play with organising the filesystem. Firstly navigate to your home directory by typing cd ~ (tilde – This should appear on the key next to 1). Typing mkdir stuff will make a directory named stuff. Change to the directory stuff and we'll create a file called hello_world by typing echo Hi this is my Atomic Linux box > hello_world. This line echo's the string 'Hi this is my Atomic Linux box' into hello_world using > (An output redirection).

    TAB key

      Cool, so so far we can switch consoles, navigate directories, and create simple text files. We shall now learn about the wonderkey. Auto-completion is a wonderful thing. Blessed be the programmer that decided they were too lazy to type the full filesystem path all the time. Repeat after me, Hail to the TAB key, the TAB key the TAB key. Hail to the TAB key, TAB key, the Man!

      Suffice to say that I'm suitably impressed by it's function in the console and as I'll demonstrate you'll know doubt be amazed also. We firstly want to make things difficult for ourselves so change directories a few times until you feel comfortably lost! Once you're happy that you are nowhere near your home directory, I want you to display the contents of the file hello_world on the console. Those that rushed ahead and are now reading this after they typed out the full path of the file will kick themselves when the learn the quicker way. If we type cat /h [TAB] you'll find that console has completed the directory /home for you. If the path doesn't complete then there are more than one options to choose from and you have to narrow the search. Hit the TAB key twice to find the output for current options on completing the path. (Note that if you only have one option the console will complete it). Once you located hello_world hit enter (You command line should look something like cat /home/sexgod/hello_world) and it should output Hi this is my Atomic Linux box.

    Locate

      Being able to display files is pretty handy but I'd say being able to find them would be better. Now most GUI's will be able to handle this quite sexily with an icon or three but where does that leave us on the command line? You could achieve this with a script using ls and grep but binaries are faster in runtime (A disadvantage is that binaries need to be recompiled every time you change them).

      The application for achieving this feat is locate. Go and try it now! Type locate hello_world and it should return /home/someone/hello_world or similar. Try a few others like gcc, glibc and linux. To get more information about any application try asking it for help. I.e. Locate --help

    What happened to my <insert device here>?

      By now we're able to navigate the system and get help if needed. It's now time to find devices and how to use them. Unix like systems don't hide the devices away from the user, there are all there ready to be found right underneath our noses.

      The directory structure that you're looking at is called Filesystem Hierarchy Standard. Everything in the system can be accessed through the file tree. Yes, Everything! From your normal files and directories, to all your devices attached to the system and even the code the kernel is currently processing. Let me quickly demonstrate the power of this filesystem, cat /proc/cpuinfo and you should be presented with detailed information about the current status of your CPU(S). You can try some others, I'll just keep rambling and you can catch up to me later, Just don't cat /proc/kcore. Don't say I didn't warn you!

      First lets find your CD devices. Devices are found under the /dev directory. For example the physical link to your primary hard drive is represented by /dev/hda and the first logical partition on that hard drive is /dev/hda1. It's not rocket science! A standard convention is to link /dev/cdrom to the first cdrom drive found on the system and is nearly always a link to /dev/cdroms/cdrom0. Now that we've located the CD-ROM device we can mount it. Come on, focus people! Can we keep our minds out of the gutter. Mounting a device simply means initialise the device and prepare it for data transfer.

      By typing the command mount -t iso9660 /dev/cdrom /mnt/cdrom we are asking the kernel to examine the device /dev/cdrom which has a filesystem of type (-t iso9660) iso9660 to be mounted in the filesystem at the point /mnt/cdrom. In English that means make the files on my CD-ROM readable from the /mnt/cdrom directory. Ensuing that you have a CD-ROM in the drive this command should execute without error. When we're finished reading (/writing) to a filesystem we also need to unmount it. It's like putting our toys away once we're finished playing with them. Because it's already mounted we can be lazy and get away with typing umount /mnt/cdrom and as long as it isn't in use (Including if you are currently in the directory) the filesystem will be unmounted. There are some workarounds for stubborn mounts and failed hardware without cutting the power but to find out more just man mount.

      For common and regularly mounted filesystems there is a file you can edit that will help cut down some time and also allow normal (Non root) users to mount the filesystem. Fire up a text editor and edit the fstab (file system table), nano /etc/fstab. You should be presented with a text config file that resembles

      /dev/hda1 /boot ext3 noauto,noatime 1 1

      /dev/hda3 / ext3 noatime 0 0

      /dev/hda2 none swap sw 0 0

      /dev/cdrom /mnt/cdrom iso9660 noauto,ro,user 0 0

      # NOTE: The next line is critical for boot!

      proc /proc proc defaults 0 0

      I'm just going to run over this real fast as there is a wealth of information on the topic (You guessed it! You can man /etc/fstab for details). The first column is the device you want to mount, the second where in the file tree you want to mount it and the third is the partition type. The fifth and sixth field relate to the program dump and fsck respectively and as such I won't cover them here. I am however interested in talking about the fourth field.

      This is a comma seperated list of options that can enable and disable features. You'll note that on the /dev/cdrom line there is noauto, ro and user. Noauto controls whether or not the filesystem is loaded at boot time (Or by a mount -a call) the deafult is auto. The next option ro satnds for read only and can be substituted for rw which is the default. The most important to use for a cdrom is user as this stipulates whether or not a normal user is able to mount this device (Mounting privileges are reserved by root). If you also have a legacy floppy drive then adding the correct details here will enable users to mount floppy drive volumes.

      This just about does it for a quick summary on how to read a CD and read write to floppies. Writing to a CD will take a little more than 10 minutes to explain so lets move on to a brief of how to get a printer going.

      The first stop on adding your printer is finding drivers for the Linux platform. Point your browser at www.linuxprinting.org and search for your printer. If you find that you have a printer thats not supported then the next stop is over to the printer HOW-To. A supported printer is a snap to setup, just follow the quick install guide for use with the appropriate printing layer. Confused? Thats fair, basically think of it as the application you call when you need to output to printer. It's similar to the difference between the driver that runs you display (Nvidia detonators for example) and the program that outputs to it (Say XFree86 as it drives the displaybuffer). We suggest using CUPS as it is held as being the most compatible.
